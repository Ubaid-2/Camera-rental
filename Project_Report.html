Phase 1: Project Proposal

1. Project Title

LensLocker - A Peer-to-Peer Camera Rental Marketplace

2. Problem Statement

In the current media industry, high-quality photography and cinematography equipment is prohibitively expensive for students, hobbyists, and early-career professionals. Simultaneously, many equipment owners possess expensive gear that remains idle for long periods, generating no value. There is currently no trusted, centralized local platform to facilitate secure, verified rentals between these two groups, forcing users to rely on unorganized and insecure social media groups.

3. Proposed Solution

LensLocker is a localized web application designed to bridge this gap. It provides:

  • For Renters: Affordable access to professional gear (Sony, Canon, Blackmagic) on a daily rental basis.
  • For Sellers: A secure platform to monetize idle assets with a verification system.
  • Security: Integration of CNIC-based identity verification and secure image handling.

4. Technology Stack

  • Frontend: HTML5, CSS3 (Glassmorphism), Vanilla JavaScript (ES6+).
  • Backend: Supabase (PostgreSQL Database, Authentication, Edge Functions).
  • Storage: Supabase Storage Buckets (for secure document handling).
  • Tools: VS Code, Git/GitHub.

Phase 2: Requirements & SRS

1. Functional Requirements

Authentication Module

  • Users must be able to Register as Buyer or Seller.
  • Registration must require Identity Verification (CNIC Front, Back, User Photo).
  • Login must be restricted for users with "Pending" status.

Seller Module

  • Create Listing: Upload camera details (Model, Price, Description) and images.
  • Dashboard: View active listings and rental request status.
  • Approval System: Approve or Reject incoming rental requests from buyers.

Buyer Module

  • Marketplace: Browse available cameras in a responsive grid layout.
  • Rental Request: Select rental dates and submit a booking request.
  • Visual Inspection: View high-quality images of the product before renting.

2. Non-Functional Requirements

  • Security: All user documents (CNIC) must be stored in private storage buckets with restricted access policies.
  • Responsiveness: The UI must adapt seamlessly to Mobile (375px) and Desktop (1920px) screens.
  • Performance: Application should load initial content in under 2 seconds.

Phase 3: System Design

1. Architectural Design

The system utilizes a Serverless / BaaS (Backend-as-a-Service) Architecture. The frontend is a decoupled Single Page Application (SPA) served via static hosting, communicating directly with the Supabase API tier.

2. Database Design (Schema)

The database is built on PostgreSQL. Below is the relational schema design.

Table: Profiles

Column Type Description
id UUID Primary Key, Links to Auth User
role Text 'buyer' or 'seller'
status Text 'pending', 'approved'
cnic_urls Text Secure paths to ID docs

Table: Cameras

Column Type Description
id UUID Primary Key
owner_id UUID Foreign Key (Profiles)
price_per_day Numeric Rental Rate (PKR)
available Boolean Availability Status

Table: Rentals

Column Type Description
id UUID Primary Key
camera_id UUID Foreign Key (Cameras)
buyer_id UUID Foreign Key (Profiles)
status Text Transaction Status

4. API Interface Design

Our application interacts with the Supabase API. Below are the key API endpoints (Table Interactions) used in the application.

4.1 User Profiles API

Method Table/Function Description
GET profiles Fetch user details (role, name, avatar).
POST auth.signUp() Register new user and trigger profile creation.
UPDATE profiles Update verification status (Admin only).

4.2 Listings API

Method Table/Function Description
GET cameras Fetch available cameras (filtered by available=true).
POST cameras Create a new camera listing (Seller only).
GET storage.from('camera_images') Generate Signed URLs for private/secure image access.

4.3 Rentals API

Method Table/Function Description
POST rentals Create a new rental request (Buyer).
UPDATE rentals Approve/Reject rental request (Seller).

2. Code Snippet: Secure Image Handling

The following function demonstrates how we secure user privacy by generating signed URLs for sensitive images on demand, rather than exposing public links.

async function loadSecureImages(path) {
    const { data } = await supabase
        .storage
        .from('cnic_images')
        .createSignedUrl(path, 3600); // URL valid for 1 hour
    return data.signedUrl;
}
